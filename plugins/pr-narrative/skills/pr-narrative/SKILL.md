# PR Narrative

## Overview

Generate a clear, human PR title and description from git history and code changes, then create or update the PR on GitHub immediately — no confirmation step.

## Invocation

```
/pr-narrative [<pr-url-or-number>]
```

- No argument → inspect current branch, find or create a PR
- With PR URL or number → update that specific PR's title and description

## Phase 0: Determine Mode

**With argument:**
- Parse PR number from the argument (handle both full URL and bare number)
- Fetch the PR using `gh pr view <number>`
- Mode: **UPDATE**

**Without argument:**
- Run `gh pr view --json number,title,url 2>/dev/null` on the current branch
- PR found → Mode: **UPDATE**
- PR not found → Mode: **CREATE**

## Phase 1: Read Context

Run all of these:

```bash
git log main..HEAD --oneline          # commit history on this branch
git diff main..HEAD --stat            # files changed summary
git diff main..HEAD                   # full diff
```

Also check for project context:
- `CLAUDE.md` — conventions, tech stack, PR format preferences
- `README.md` — project description if CLAUDE.md absent

## Phase 2: Detect Change Type

Scan the diff for signals:

| Signal | Detected change type |
|--------|---------------------|
| `.tsx`, `.jsx`, `.vue`, `.css`, `.scss` files changed | Frontend / UI |
| `migrations/`, `schema.rb`, `prisma/` changed | Database |
| `.github/workflows/`, `Dockerfile`, `*.yml` changed | Infrastructure |
| `*_test.*`, `*.spec.*`, `test_*.py` changed | Tests |
| `README`, `docs/`, `*.md` changed | Documentation |

## Phase 3: Generate Title + Description

**Title rules:**
- Imperative voice, present tense ("Add X", "Fix Y", "Remove Z")
- Under 70 characters
- No ticket numbers unless found in commit messages
- Specific — describe what changed, not that something changed
- If `CLAUDE.md` specifies a title format, follow it

**Description template:**

```markdown
## Why
<1–2 sentences on the motivation — what problem this solves or what feature this enables>

## What changed
- <bullet summarizing the key change>
- <bullet for each meaningful change, max 5 bullets>

## How to test
<numbered steps a reviewer should follow to verify this works>

## Screenshots
<!-- Add screenshots here if UI changes -->
```

- Include **Screenshots** section only if frontend change detected
- Omit it for backend-only, infra, or docs PRs
- Keep bullets focused on *what* changed and *why it matters*, not line-by-line code description
- "How to test" should be specific steps, not "run the tests"

## Phase 4: Apply

**UPDATE mode:**
```bash
gh pr edit <number> --title "<generated-title>" --body "<generated-description>"
```

**CREATE mode:**
```bash
gh pr create --title "<generated-title>" --body "<generated-description>"
```

After applying, output the PR URL so the user can open it directly.

## Rules

- Always replace the full title and description — never append to existing content
- If `git diff main..HEAD` is empty, tell the user there are no changes on this branch vs main and stop
- If `gh` CLI is not available or not authenticated, tell the user and print the generated title + description as plain text instead so they can paste it manually
- If the branch has only one commit, the title should closely reflect that commit message (cleaned up)
- Never include "Generated by AI" or any tool attribution in the PR title or description
- Keep the description under 1000 characters total where possible — reviewers skim
